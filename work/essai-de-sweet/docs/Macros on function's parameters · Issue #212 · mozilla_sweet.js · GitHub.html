amatiasq commented 2 months ago

I'm trying to write function parameter's type using C++ like notation

function Something(String signal, Number value) { }

To produce

function Something(signal, value) { }

So I create a function macro and a helper for the parameters

macro params_helper {
  rule { $type:ident $name:ident , $rest ... } => {
    $name , params_helper $rest ...
  }
  rule { $name:ident , $rest ... } => {
    $name , params_helper $rest ...
  }
  rule { $type:ident $name:ident } => { $name }
  rule { $name:ident } => { $name }
}

let function = macro {
  rule { $name ($params ...) $body } => {
    function $name(params_helper $params ...) $body
  }
}

function Something(String signal) { }

And it produces this result: Test it

function Something$5523(params_helper$5524, String$5525, signal$5526) { }

I don't understand why a comma is placed here but anyway I try to move function to a helper so params_helper is parsed but it produces the same result since function_helper is parsed before params_helper. Test it

I tried to add parentheses as @disnet says on #37 but an error is produced since commas are placed between identifiers. Test it

Finally I tried to simplify the issue in the minimal code:

macro params_helper {
  rule { $type:ident $name:ident } => { $name }
}
(function(params_helper String signal) { })

And this produces

(function (params_helper$835, String$836, signal$837) { });

So I see two issues here:

First, commas are added between identifiers on function arguments, I don't understand why this is happening but looks reported at #63 is there a plan to fix it?

Second, function parameters macros are not expanded. I see some issues related to this #37, #103 and #120, but there isn't any way to do it? no work around? not even with procedural macros?

It annoys me because since the first time I heard about sweet.js I thought the main possibilities were on function arguments. Anyway I think it's an amazing job and I'm sure it will be part of tomorrow's day to day javascript.

macro params_helper {

macro real_params_helper {
  rule { $param ... , $rest ... } => {

  }
  rule { $param ... } => {
    single_param $param ...
  }

Thanks a lot for your help @natefaubion

I think function parameters are where sweet.js contributes more value because on simple sentences I can create functions:

class("Something", {
  member: function() { ... }
});

Or even for documentation there are some hacks we can do, they aren't nice but they fulfill a need.

class("Something", {
  member: "Does bla bla bla bla..."
  member: function() { }
});

And operations like variable swap similar things aren't things I do day to day.

But every day, doesn't matter what I'm doing I'm always writing functions and it's parameters and I don't usually do it by hand but it would be nice to have some type of validation but they are from my point of view the more static part of javascript, you cannot invoke a function to add functionality and if you hide information on it's name you have to repeat it each time you use the parameter:

function Something(String_foo, opt_bar) {
  if (opt_bar)
    return String_foo;
}

function Something(sFoo, oBar) { // oBar means "object" or "optional"?
  if (oBar)
    return sFoo;
}

The closest I've always found to extend parameters functionality was creating a object which contains parsed arguments (on runtime):

function Something(String_foo, opt_bar) {
  // we pass Something so it can .toString() it and watch the arguments names
  var args = parseArgs(Something, arguments);
  if (args.bar)
    return args.foo;
}

Or use block comments:

function Something(/*String*/ foo, /*optional*/, bar) { }

As I write parameters every time this options are too heavy.

I think it's important to document code from inside and javascript parameters were always a impediment for this so when I heard what sweet.js can do the first thing than came to my mind was "finally, parameters can be extended".

natefaubion commented 2 months ago

Cool! I think it would be really difficult to make a proper static type system like typed clojure, though you could take the type parameters and turn them into comment docs on the function, which you could feed into the closure compiler. Sweet does its best to preserve comments on the read tree, so you can actually add to it in case macros. Try adding a comment before a function, and then inspect the token. You'll see a leadingComments attribute.
